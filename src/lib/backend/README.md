# Backend frontier

This folder is the **only** part of the app that knows which backend
provider we use. The rest of the codebase imports from
`@/lib/backend`, never from `@/integrations/supabase/client`.

## Why this exists

InfoSec / vendor portability requirement: if Verisure decides to drop
Supabase (or Lovable Cloud) for any other provider, all changes happen
inside this directory. No business code outside `src/lib/backend/` may
import the Supabase SDK directly.

## What is portable vs. Supabase-shaped

| Surface           | Portable?       | Notes                                                |
|-------------------|-----------------|------------------------------------------------------|
| `auth.*`          | YES             | Stable interface (`signIn`, `signOut`, `getSession`, `onAuthStateChange`, `getUser`). Reimplement on any provider that exposes JWT-based password auth. |
| `rpc(name, args)` | YES             | Maps to any provider that supports server-side stored procedures or remote functions. |
| `invokeFunction`  | YES             | Maps to any serverless / edge function platform.     |
| `db` (query builder) | **NO**       | Returns the raw PostgREST query builder. Replacing the backend with a non-PostgREST provider (Firestore, Hasura, custom REST) requires re-implementing every `db.from('table').select()` call. We accept this trade-off: rewriting 67 query sites every time we want to switch providers is unrealistic; rewriting them once during a real migration is. |

## How to migrate to a different backend

1. Add a new adapter file (e.g. `adapter.firebase.ts`) implementing the
   `BackendAdapter` interface in `types.ts`.
2. Switch the export in `index.ts` based on `VITE_BACKEND_PROVIDER`
   (currently hardcoded to `supabase`).
3. For the `db` surface, do a controlled migration: each call site needs
   to be re-expressed in the new provider's idiom. There is no shortcut.
4. Re-deploy edge functions on the new platform. The function names
   exposed via `invokeFunction` should stay stable as a contract.

## Files

- `index.ts` — public entry point. Imports outside this folder use only this.
- `types.ts` — `BackendAdapter` interface and shared types.
- `adapter.supabase.ts` — current Supabase implementation.
- `auth.ts` — provider-agnostic auth wrapper exposed by the adapter.

## Forbidden imports (lint-enforceable)

Outside this folder:
- `from '@supabase/supabase-js'` — never
- `from '@/integrations/supabase/client'` — never (use `@/lib/backend`)

The two `src/integrations/supabase/{client,types}.ts` files remain
because they are auto-generated by the Supabase CLI and consumed by the
adapter. They are not the public API.
